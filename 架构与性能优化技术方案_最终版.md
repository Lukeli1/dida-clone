## 一、当前架构诊断

### 1.1 核心问题

`src/App.tsx` 是一个 **1767 行的巨型组件**，承载了全部状态和业务逻辑：

| 问题 | 具体表现 | 影响 |
|------|----------|------|
| 状态集中 | 23 个 `useState` + 3 个 `useRef` 集中在 App.tsx | 组件难以维护，状态关系复杂 |
| 面条代码 | 所有 CRUD、筛选、排序、拖拽逻辑都在一个文件 | 修改任何功能都需要理解全文件 |
| 无状态管理 | 数据全靠 props 逐层传递 | 深层组件获取数据成本高 |
| 无缓存层 | 每次操作都走 `api` 直接调用后端 | 重复请求、无离线能力 |
| 无虚拟列表 | 全部任务一次性渲染 | 任务量>500 时滚动卡顿 |
| localStorage 孤岛 | 习惯打卡/番茄钟数据存 localStorage，不走 api 层 | 数据层不统一，无法纳入全局状态 |
| 硬编码密钥 | `llm.ts:15` 硬编码了 API Key | 安全隐患 |

### 1.2 数据流现状

```
┌─────────────────────────────────────────────────┐

### 1.3 关键发现：localStorage 数据孤岛

以下两个组件**不走 `api.ts`**，直接读写 `localStorage`，是现有架构中的数据孤岛：

| 组件 | localStorage Key | 数据内容 |
|------|-----------------|----------|
| `HabitView.tsx` | `habits_data` | 习惯列表 + 打卡记录 |
| `PomodoroView.tsx` | `pomodoro_settings` | 番茄钟设置 |
| `PomodoroView.tsx` | `pomodoro_stats` | 番茄钟统计 |

**影响**: Zustand store 和 React Query 都无法管理这些数据。重构时必须单独处理。

### 1.4 关键发现：api.ts 浏览器降级机制

`api.ts` 中每个方法都有 `if (!isTauri)` 分支，使用模块级 mock 数组（`mockTasks`、`mockLists` 等）模拟后端。这些 mock 数组是**可变内存状态**。

**对 React Query 的影响**: 
- `queryFn: api.getTasks` 在非 Tauri 环境下返回 mock 数组的浅拷贝
- mock 数据被其他操作修改后，React Query 的 `staleTime` 缓存可能返回旧数据
- **结论**: React Query 迁移时必须考虑这个降级路径，或决定移除 mock 模式

### 1.5 关键发现：loadData() 的 8 处调用点

`loadData()` 是 App.tsx 中的全局数据重载函数，在以下 8 处被调用：

| # | 调用位置 | 函数名 | 行号(约) | 触发条件 |
|---|----------|--------|----------|----------|
| 1 | line 83 | useEffect (auto-archive) | 自动归档完成后 | tasks 首次加载 |
| 2 | line 391 | handleToggleTask | 重复任务完成后 | result.new_task_id 存在 |
| 3 | line 601 | handleReorder | 拖拽排序失败后 | catch 分支 |
| 4 | line 678 | handleBatchComplete | 批量完成失败后 | catch 分支 |
| 5 | line 694 | handleBatchDelete | 批量删除失败后 | catch 分支 |
| 6 | line 709 | handleBatchPriority | 批量优先级失败后 | catch 分支 |
| 7 | line 724 | handleBatchMoveList | 批量移动失败后 | catch 分支 |
| 8 | line 740 | handleBatchArchive | 批量归档失败后 | catch 分支 |

迁移到 React Query 后，这 8 处全部需要替换为 `queryClient.invalidateQueries({ queryKey: queryKeys.tasks.all })`。

- ❌ 不改变数据库 schema（SQLite 表结构保持不变）

- ❌ 不迁移习惯/番茄钟的 localStorage 数据到 SQLite（保留 localStorage 方案）
- ❌ 不改变 Tauri 后端 Rust 代码
- ❌ 不移除 `api.ts` 的浏览器降级 mock 机制（保留用于开发调试）

### 5.1 目标

```

### 5.3 错误处理原则

**关键**: Zustand store 是纯 JavaScript 对象，**不能调用 React Hooks**（如 `useToast`）。

Store 中的错误处理采用两层策略：
1. **store 内部**: `console.error` 记录错误
2. **组件层**: 通过订阅 store 状态变化或 action 返回值，由组件负责调用 `toast`

```typescript
// store 中：返回操作结果，不直接调用 toast
createTask: async (req): Promise<Task | null> => {
  try {
    const newTask = await api.createTask(req)
    set((state) => ({ tasks: [newTask, ...state.tasks] }))
    return newTask  // 返回结果，组件层根据 null 判断是否 toast.error
  } catch (error) {
    console.error('Failed to create task:', error)
    return null
  }
},

// 组件中：根据返回值决定 toast
const handleCreateTask = async () => {
  const result = await useTaskStore.getState().createTask(req)
  if (result) {
    toast.success('任务已创建')
  } else {
    toast.error('创建任务失败')
  }
}
```

#### `src/stores/taskStore.ts`

```typescript
import { create } from 'zustand'
import type { Task, CreateTaskRequest } from '../types'
import { api } from '../api'

interface TaskState {
  tasks: Task[]
  loading: boolean
  setTasks: (tasks: Task[]) => void
  loadTasks: () => Promise<void>
  createTask: (req: CreateTaskRequest) => Promise<Task | null>
  updateTask: (id: number, updates: Partial<Task>) => Promise<boolean>
  deleteTask: (id: number) => Promise<boolean>
  toggleTask: (task: Task) => Promise<{ success: boolean; newTaskGenerated: boolean }>
  reorderTasks: (items: { id: number; sort_order: number }[]) => Promise<boolean>
  moveTask: (taskId: number, newDate: string) => Promise<boolean>
  reloadAll: () => Promise<void>  // 替代 loadData() 的部分功能
}

export const useTaskStore = create<TaskState>((set, get) => ({
  tasks: [],
  loading: true,

  setTasks: (tasks) => set({ tasks }),

  loadTasks: async () => {
    set({ loading: true })
    try {
      const data = await api.getTasks()
      set({ tasks: data, loading: false })
    } catch (error) {
      console.error('Failed to load tasks:', error)
      set({ loading: false })
    }
  },

  createTask: async (req) => {
    try {
      const newTask = await api.createTask(req)
      set((state) => ({ tasks: [newTask, ...state.tasks] }))
      return newTask
    } catch (error) {
      console.error('Failed to create task:', error)
      return null
    }
  },

  updateTask: async (id, updates) => {
    try {
      await api.updateTask(id, updates)
      set((state) => ({
        tasks: state.tasks.map((t) =>
          t.id === id ? { ...t, ...updates, updated_at: new Date().toISOString() } : t
        ),
      }))
      return true
    } catch (error) {
      console.error('Failed to update task:', error)
      return false
    }
  },

  deleteTask: async (id) => {
    try {
      await api.deleteTask(id)
      set((state) => ({
        tasks: state.tasks.filter((t) => t.id !== id),
      }))
      return true
    } catch (error) {
      console.error('Failed to delete task:', error)
      return false
    }
  },

  toggleTask: async (task) => {
    try {
      if (!task.completed && task.repeat_rule) {
        const result = await api.completeTask(task.id)
        set((state) => ({
          tasks: state.tasks.map((t) =>
            t.id === task.id
              ? { ...t, completed: true, updated_at: new Date().toISOString() }
              : t
          ),
        }))
        if (result.new_task_id) {
          await get().loadTasks()
          return { success: true, newTaskGenerated: true }
        }
        return { success: true, newTaskGenerated: false }
      }
      await api.updateTask(task.id, { completed: !task.completed })
      set((state) => ({
        tasks: state.tasks.map((t) =>
          t.id === task.id ? { ...t, completed: !task.completed } : t
        ),
      }))
      return { success: true, newTaskGenerated: false }
    } catch (error) {
      console.error('Failed to toggle task:', error)
      return { success: false, newTaskGenerated: false }
    }
  },

  reorderTasks: async (items) => {
    try {
      await api.reorderTasks(items)
      const sortOrderMap = new Map(items.map((item) => [item.id, item.sort_order]))
      set((state) => ({
        tasks: state.tasks.map((t) =>
          sortOrderMap.has(t.id) ? { ...t, sort_order: sortOrderMap.get(t.id)! } : t
        ),
      }))
      return true
    } catch (error) {
      console.error('Failed to reorder tasks:', error)
      await get().loadTasks()  // 失败时重新加载
      return false
    }
  },

  moveTask: async (taskId, newDate) => {
    try {
      const task = get().tasks.find((t) => t.id === taskId)
      const updates: Partial<Task> = { due_date: newDate }
      if (task?.end_date && task?.due_date) {
        const duration =
          new Date(task.end_date).getTime() - new Date(task.due_date).getTime()
        updates.end_date = new Date(
          new Date(newDate).getTime() + duration
        ).toISOString()
      }
      await api.updateTask(taskId, updates)
      set((state) => ({
        tasks: state.tasks.map((t) =>
          t.id === taskId
            ? { ...t, ...updates, updated_at: new Date().toISOString() }
            : t
        ),
      }))
      return true
    } catch (error) {
      console.error('Failed to move task:', error)
      return false
    }
  },

  reloadAll: async () => {
    await get().loadTasks()
  },
}))
```

const reorderMutation = useMutation({
  mutationFn: api.reorderTasks,
  onMutate: async (items) => {
    // 乐观更新
    await queryClient.cancelQueries({ queryKey: queryKeys.tasks.all })

## 八、Phase 4: 长列表虚拟化（高风险）

### 8.1 目标

### 8.2 前置优化：taskTree 树构建性能

**当前代码** (`App.tsx`):
```typescript
const taskTree = useMemo(() => {
  const topLevel = filteredTasks.filter(t => !t.parent_id)
  const subtasks = filteredTasks.filter(t => t.parent_id)
  return topLevel.map(task => ({
    ...task,
    subtasks: subtasks.filter(st => st.parent_id === task.id)  // O(n*m)
  }))
}, [filteredTasks])
```

**问题**: 对每个顶层任务都 `filter` 一次全部子任务，时间复杂度 O(n*m)。

**优化方案**: 用 Map 索引替代 filter:

```typescript
const taskTree = useMemo(() => {
  const subtaskMap = new Map<number, Task[]>()
  const topLevel: Task[] = []

  for (const task of filteredTasks) {
    if (task.parent_id) {
      const arr = subtaskMap.get(task.parent_id)
      if (arr) arr.push(task)
      else subtaskMap.set(task.parent_id, [task])
    } else {
      topLevel.push(task)
    }
  }

  return topLevel.map((task) => ({
    ...task,
    subtasks: subtaskMap.get(task.id) || [],
  }))
}, [filteredTasks])
```

## 十一、统一错误处理策略

### 11.1 Tauri IPC 错误处理

当前代码中 `invoke()` 调用失败时只 `console.error` + `toast.error`，缺乏统一策略。

**建议方案**: 在 `api.ts` 中统一包装：

```typescript
// src/lib/errorHandler.ts
export class AppError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly context?: unknown
  ) {
    super(message)
  }
}

export function handleApiError(error: unknown, context: string): AppError {
  console.error(`[${context}] Error:`, error)
  if (error instanceof Error) {
    return new AppError(error.message, 'API_ERROR', context)
  }
  return new AppError('未知错误', 'UNKNOWN', context)
}
```

### 11.2 Store 层错误传播

Store action 返回 `boolean` 或 `null`，由组件层决定如何提示用户：

```typescript
// store: 返回结果
deleteTask: async (id): Promise<boolean> => {
  try {
    await api.deleteTask(id)
    set((state) => ({ tasks: state.tasks.filter((t) => t.id !== id) }))
    return true
  } catch (error) {
    console.error('Failed to delete task:', error)
    return false
  }
}

// 组件: 根据结果 toast
const handleDelete = async (id: number) => {
  const success = await useTaskStore.getState().deleteTask(id)
  if (success) {
    toast.success('任务已删除')
  } else {
    toast.error('删除任务失败')
  }
}
```
