## 二、架构现状分析

### 2.1 视图路由 —— 手工 switch/case

App.tsx 通过 `currentView` 字符串做路由分发：

```tsx
{currentView === 'calendar' && <CalendarView />}
{currentView === 'stats' && <StatsView />}
{currentView === 'settings' && <SettingsView />}
{currentView === 'ai' && <AIAssistant />}
// ... 共 8 个分支
```

- ✅ 小型项目够用，无外部依赖
- ❌ 无法懒加载（所有视图一次性打包）
- ❌ 无法独立测试某个视图
- ❌ 没有 URL 同步，Tauri 下无关紧要但限制了扩展性

### 2.2 状态管理 —— Store 干净但组件层过度订阅

**Store 层目前是项目中最干净的模块：**
- 5 个 Zustand store，相互之间无交叉引用
- filterStore 极简 (30 行)
- 每个 store 职责单一

**但组件层面的使用方式存在明显问题：**

#### 问题 1：App.tsx 一次性订阅 36 个 UI 状态

```tsx
const currentView = useUIStore(s => s.currentView)
const selectedListId = useUIStore(s => s.selectedListId)
const selectedTagId = useUIStore(s => s.selectedTagId)
// ... 共 36 个独立 selector
const filters = useFilterStore()  // 订阅了 filterStore 整体
```

- 任何一个 UI 状态变化都会触发 App.tsx 整体重新渲染
- `useFilterStore()` 无 selector 订阅了整个 store，浪费
- 订阅代码本身占了 80 行

#### 问题 2：筛选逻辑分布在两个地方

- `filterStore` 存储筛选条件（priority/dateRange/tagId/listId）
- `App.tsx` 中的 `filteredTasks` 计算逻辑既有视图级筛选又有组合筛选
- **存在功能重复**：`selectedTagId` 和 `filters.tagId` 做了两遍标签筛选
- **存在逻辑重叠**：Today 视图和 `dateRange: 'today'` 筛选条件含义相同

#### 问题 3：uiStore 领域混杂

单个 uiStore (113 行 / 19 个 action) 同时管理：
- 视图导航（currentView, selectedListId, selectedTagId）
- 显示控制（showCompleted, showOverdue, showFilters）
- 批量操作（batchMode, selectedTaskIds）
- 拖拽状态（isDraggingTask, dragOverCalendarDate）
- AI 状态（aiMode, aiParsing）
- 搜索（searchQuery）
- 子任务输入（subtaskInputs）

「高内聚低耦合」未体现。

### 2.3 组件通信 —— 深度 Props Drilling

**TaskItem.tsx (727 行)** 是问题最严重的组件：

```tsx
<TaskItem
  task={task}
  tags={tags}
  lists={lists}
  isSelected={...}
  isExpanded={...}
  onToggleExpand={...}
  subtaskInput={...}
  onSubtaskInputChange={...}
  onCreateSubtask={...}
  onToggle={...}
  onToggleSubtask={...}
  onClick={...}
  onReorder={...}
  onDelete={...}
  batchMode={...}
  isSelectedForBatch={...}
  onToggleSelect={...}
  onInlineEdit={...}
  onArchive={...}
  onDragStartGlobal={...}
  onDragEndGlobal={...}
  onSetDate={...}
  onSetPriority={...}
  onTogglePin={...}
  onToggleTag={...}
  onDuplicate={...}
  onCreateNewTag={...}
  // 合计 29 个 prop
/>
```

- 这些 props 在 App.tsx → TaskList.tsx → TaskItem.tsx 三层传递
- 每增加一个功能就要修改三处接口
- props 中有大量「透传」回调（如 `onReorder={() => {}}`）

**同一个模式在 TaskDetail.tsx (710 行)** 也存在：
传递 tags/lists/onUpdate/onDelete/onAddTag 等。

### 2.4 API 层 —— 无错误处理、无缓存、无重试

`api.ts` (264 行) 中有 16 个端点，但：

```typescript
// 每个函数长这样：
async updateTask(id: number, updates: Partial<Task>) {
  if (!isTauri) { /* mock */ }
  else { await invoke('update_task', { id, updates }) }
}
```

- ❌ `api.ts` 本身无 try/catch，异常直接向上抛
- ⚠️ 错误由 Store 层 `try/catch` 兜底，但缺少统一错误上报和用户提示
- ⚠️ `lib/errorHandler.ts` 中定义了 `AppError` 类但未被引用
- ❌ 无请求合并/去重
- ⚠️ React Query 基础设施已完备（QueryClient/staleTime/retry/queryKeys/invalidate），但 Stores 尚未接入

### 2.5 深色模式 —— !important 暴力覆盖

`index.css` 中有大量：

```css
.dark .bg-white { background-color: #1f2937 !important; }
.dark .text-gray-900 { color: #f3f4f6 !important; }
.dark .border-gray-200 { border-color: #374151 !important; }
```

- 共约 40 行 `!important` 规则
- 每次新增组件都要确保不会漏掉覆盖
- Tailwind 3.4 原生支持 `dark:` 前缀，完全不需要 `!important`
- 维护成本高，新加入的开发者容易遗漏

### 2.6 标签/清单管理的本地状态不一致

`tagStore` 中的 `addTagToTask` / `removeTagFromTask` 只调 API，不更新本地 tags 状态：

```typescript
addTagToTask: async (taskId, tagId) => {
  await api.addTagToTask(taskId, tagId)
  // 没有更新 set 或重新 loadTasks()
}
```

调用方必须手动触发 `taskStore.loadTasks()`，容易忘记。

### 2.7 代码目录结构 —— 平铺无分层

```
src/
├── App.tsx                # 1538 行，包括所有视图逻辑
├── api.ts                 # 16 个端点
├── types.ts               # 92 行，所有类型
├── index.css              # 全局样式 + 深色模式覆盖
├── main.tsx               # 入口
├── components/            # 18 个文件平铺
│   ├── Sidebar.tsx        # 619 行
│   ├── TaskItem.tsx       # 727 行
│   └── ...                # 其他视图组件
├── stores/                # 5 个文件
├── utils/                 # 4 个文件
└── lib/                   # 3 个文件
```

- views 和 shared components 混在一起
- 没有按功能域分组
- 没有 hooks/ 目录放置自定义 hook

### 2.8 零测试覆盖

- 无单元测试
- 无组件测试
- 无 E2E 测试
- 纯手动测试

## 三、改进方案

按优先级从高到低排列。

### P0：紧急/高频影响

#### 方案 1：App.tsx 拆分为独立视图 Hook + 按需渲染

**现状**：1538 行，36 个 store 订阅，筛选/排序/创建/删除逻辑全部在顶层

**改进**：

```typescript
// 抽离 4 个自定义 Hook
src/hooks/
├── useTaskFiltering.ts    // 筛选 + 排序 + 树组装逻辑（从 App.tsx 搬出）
├── useTaskActions.ts      // 创建/更新/删除/拖动/置顶等 handler
├── useKeyboardShortcuts.ts // 键盘快捷键（从 App.tsx 搬出）
└── useAppInit.ts         // 启动初始化 + 自动归档 + 桌面通知
```

**效果**：
- App.tsx 从 1538 行 → ~300 行
- 筛选逻辑可独立测试
- 每个 hook 可独立订阅 store，粒度精确

**涉及文件**：
- 新增 `src/hooks/useTaskFiltering.ts`
- 新增 `src/hooks/useTaskActions.ts`
- 新增 `src/hooks/useKeyboardShortcuts.ts`
- 新增 `src/hooks/useAppInit.ts`
- 修改 `src/App.tsx`

---

#### 方案 2：合并 App.tsx 中的筛选逻辑与 filterStore

**现状**：两套筛选并行，`filteredTasks` 有 6 个分支 + 再次叠加 filterStore

**改进**：创建一个统一的 `useFilteredTasks` hook 替代 App.tsx 中的 inline 计算：

```typescript
export function useFilteredTasks(tasks: Task[]) {
  const currentView = useUIStore(s => s.currentView)
  const selectedListId = useUIStore(s => s.selectedListId)
  const selectedTagId = useUIStore(s => s.selectedTagId)
  const searchQuery = useUIStore(s => s.searchQuery)
  const filters = useFilterStore(s => ({ priority: s.priority, dateRange: s.dateRange }))

  return useMemo(() => {
    // 统一的筛选逻辑，消除重复
  }, [tasks, currentView, selectedListId, selectedTagId, searchQuery, filters])
}
```

**效果**：
- 消除标签/清单筛选的双重过滤
- 逻辑集中，便于维护和单测
- filterStore 和视图筛选变成「或」关系而非「叠加」关系

**涉及文件**：
- 新建 `src/hooks/useFilteredTasks.ts`
- 修改 `src/App.tsx`（删除 inline filteredTasks 逻辑）
- 可考虑修改 `src/stores/filterStore.ts`

#### 方案 3：TaskItem / TaskDetail 引入 Props 收敛

**现状**：TaskItem 29 个 prop，TaskDetail 也有大量 props

**改进**：将高频回调聚合到 Context 中：

```typescript
// 方案 A：TaskActionContext
const TaskActionContext = createContext<{
  onToggle: (task: Task) => void
  onDelete: (id: number) => void
  onArchive: (id: number) => void
  onSetDate: (id: number, date: string) => void
  onSetPriority: (id: number, priority: number) => void
  onTogglePin: (id: number) => void
  onToggleTag: (id: number, tagId: number) => void
  onDuplicate: (id: number) => void
  onCreateNewTag: (name: string) => void
  |null>()
>()

// 在 App.tsx 中包裹
<TaskActionContext.Provider value={...}>
  <TaskItem ... />
</TaskActionContext.Provider>

// TaskItem 中直接 useContext
const { onToggle, onDelete, ... } = useContext(TaskActionContext)
```

**效果**：TaskItem props 从 26 个降至 6-8 个（仅 task/lists/tags/isSelected 等数据 prop），回调全部走 Context。

**涉及文件**：
- 新建 `src/contexts/TaskActionContext.ts`
- 修改 `src/App.tsx`（添加 Provider）
- 修改 `src/components/TaskItem.tsx`
- 修改 `src/components/TaskList.tsx`
- 可选：修改 `src/components/TaskDetail.tsx`

#### 方案 5：将 Stores 接入已有的 React Query 基础设施

**现状**：@tanstack/react-query 的 QueryClient / staleTime / retry / queryKeys / invalidate 已全部配置好，但 taskStore 仍直接调 `api.getTasks()` 管理数据

**改进**：用 React Query 替代直接调用 api.ts：

```typescript
// 不是替换 api.ts，而是在组件/store 中改用 useQuery
// 当前：
const tasks = await api.getTasks()
setTasks(data)

// 改进后：
const { data: tasks, isLoading, error } = useQuery({
  queryKey: ['tasks'],
  queryFn: api.getTasks,
  staleTime: 30_000,     // 30s 内不走网络
  gcTime: 5 * 60_000,    // 5min 缓存
})
```

**效果**：
- 零改动获得缓存/去重/重试（QueryClient 已配置 staleTime=30s + retry=1）
- 消除 tagStore 本地状态不一致问题（改用 query invalidation 代替手动 reloadTasks）
- Tauri 本地调用几乎零延迟，staleTime 可进一步拉长

**涉及文件**：
- 修改 `src/main.tsx`（添加 QueryClientProvider）
- 修改 `src/stores/taskStore.ts`（可选择逐步迁移）
- 需要讨论是否直接替换 store 还是渐进式接入

#### 方案 6：深色模式改用 Tailwind dark: 前缀

**现状**：40 行 `!important` 覆盖

**改进**：逐文件将 `.dark .bg-white {}` 改为组件中的 `dark:bg-gray-800 bg-white`

**效果**：
- 消除 `!important` 带来的优先级问题
- 组件级别的深色样式更精确
- 新组件开箱即用

**工作量**：需修改约 18 个组件，每个组件平均 10-20 处 `bg-gray-50` / `bg-white` / `text-gray-900` / `border-gray-200` 等。

**建议**：分批进行，先从核心组件（Sidebar / TaskItem / TaskDetail）开始。

**涉及文件**：全部 18 个组件 + index.css（删除深色覆盖部分）

#### 方案 7：tagStore 本地状态同步

**现状**：`addTagToTask`/`removeTagFromTask` 不更新本地

**改进**：

```typescript
addTagToTask: async (taskId, tagId) => {
  await api.addTagToTask(taskId, tagId)
  const { tasks, setTasks } = get()
  setTasks(tasks.map(t => {
    if (t.id !== taskId) return t
    return {
      ...t,
      tag_ids: [...(t.tag_ids || []), tagId]
    }
  }))
}
```

**效果**：无需手动 `reloadTasks()`，减少一次全量刷新。

**涉及文件**：
- 修改 `src/stores/tagStore.ts`
- 修改 `src/stores/taskStore.ts`（可能需要暴露局部更新方法）

## 六、总结

当前项目最核心的三个架构问题：

1. **App.tsx 过度膨胀** — 1538 行主管道，是所有功能的「单点故障」，优先解决
2. **TaskItem prop drilling** — 29 个 prop 透传三层，每次改接口代价高
3. **筛选逻辑分散** — 两套过滤机制并存，容易产生歧义和重复

这三项 P0 问题互不阻塞，可以并行推进。建议先从 **App.tsx 拆 hook** 入手，因为它最有「立即见效」的成就感，同时为后续 Context 收敛和筛选统一铺平道路。
