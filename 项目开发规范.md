# 滴答清单复刻 项目开发规范

## 一、双模型协作策略

### 1.1 模型分工

| 模型层级 | 可用模型 | 职责 | 类比角色 |
|---------|---------|------|---------|
| **贵模型** | DeepSeekV4Pro / GLM5.2 | 架构设计、复杂逻辑、代码评审、疑难 Bug 排查 | 架构师 + Code Reviewer |
| **便宜模型** | Step7Flash / DeepSeekV4Flash | 按方案写代码、样式批量修改、文档更新、简单重构 | 执行程序员 |

### 1.2 核心原则

> 贵的做决策，便宜的做执行。

- **不要让便宜模型做架构决策。** 反例：直接对便宜模型说"帮我加个自动启动功能"——它会硬编码一个 `useState` 糊弄，没有后端调用。
- **正确做法**：先用贵模型产出方案（改哪些文件、用什么库/插件、影响面评估），再把方案交给便宜模型执行具体代码修改。
- **改完后用贵模型做一次快速 review**，确认没有引入新问题。

### 1.3 场景分工表

| 场景 | 用哪个 | 理由 |
|------|--------|------|
| 新功能架构设计 | 贵 | 需要理解现有架构、权衡方案取舍 |
| 按方案写代码实现 | 便宜 | 设计已定，就是翻译成代码 |
| 复杂 Bug 排查 | 贵 | 需要追溯逻辑链、理解多文件联动 |
| 深色模式/样式批量修改 | 便宜 | 机械替换，规则明确 |
| 代码重构（抽 Hook、拆组件） | 贵 | 需要判断边界和影响面 |
| 写单元测试 | 便宜 | 给定接口，覆盖边界即可 |
| Code Review | 贵 | 需要发现潜在问题、安全隐患 |
| README/文档更新 | 便宜 | 事实性总结，不需深度推理 |
| Tauri 命令/插件接入 | 贵 | 涉及 Rust FFI、权限、配置联动 |
| Tailwind 样式微调 | 便宜 | 规则明确，只需添加 `dark:` 前缀 |

### 1.4 推荐开发流程

```
需求 → 贵模型分析（问题定位 + 方案设计 + 影响面评估）
         ↓
     便宜模型执行（按方案修改代码）
         ↓
     贵模型快速 Review（确认无新问题）
         ↓
     构建验证 → 提交推送
```

## 二、版本管理

### 2.1 SemVer 规范

遵循严格的语义化版本：

| 级别 | 触发条件 | 示例 |
|------|---------|------|
| **Major** | 不兼容的破坏性改动、架构大重构 | API 签名变更、数据库 Schema 破坏性修改 |
| **Minor** | 向下兼容的新功能添加 | 新增模块、新增视图、新增设置项 |
| **Patch** | 向下兼容的问题修复 | Bug 修复、样式微调、文档更新 |

### 2.2 版本号同步

**每次发版必须同步 3 个文件：**

| 文件 | 路径 |
|------|------|
| `package.json` | 项目根目录 |
| `tauri.conf.json` | `src-tauri/` |
| `Cargo.toml` | `src-tauri/` |

> ⚠️ 历史教训：`tauri.conf.json` 和 `Cargo.toml` 多次滞后于 `package.json`。

### 2.3 发版流程

1. 功能开发完成
2. 评估变更级别（Major / Minor / Patch）
3. 将建议版本号发给用户确认
4. 用户确认后，更新 3 个版本文件
5. 提交并推送至 GitHub

## 三、Git 操作规范

### 3.1 每次操作前

- **必须先 `git pull`** 确保在最新代码基上工作

### 3.2 凭据问题

- Windows 环境下，bash 中的 `git pull/push` 因无法访问 Windows Credential Manager 会失败
- **解决方案**：`GIT_ASKPASS=echo git pull` / `GIT_ASKPASS=echo git push`

### 3.3 提交规范

- 使用中文 commit message，格式：`类型: 描述`
- 类型：`feat`（新功能）、`fix`（修复）、`chore`（杂项）、`docs`（文档）、`refactor`（重构）

## 四、代码规范

### 4.1 Tailwind 深色模式

- 项目使用 `darkMode: 'class'` 策略
- 所有组件必须同时考虑亮色和深色模式
- 映射规则：
  - `bg-white` → `dark:bg-gray-900`
  - `bg-gray-50` → `dark:bg-gray-800`
  - `border-gray-200` → `dark:border-gray-700`
  - `text-gray-500` → `dark:text-gray-400`
  - `text-gray-700` → `dark:text-gray-200`
  - `hover:bg-gray-100` → `dark:hover:bg-gray-700`

### 4.2 状态管理

- 使用 Zustand v5，共 6 个独立 Store
- Handler 函数使用 `getState()` 模式避免闭包过时
- 跨 Store 同步通过动态 import 实现

### 4.3 组件 Props

- 复杂组件优先使用 Context 收敛回调（如 TaskActionContext）
- 避免 Props 过多（参考 TaskItem 从 29 个精简到 6 个）

### 4.4 API 层

- 统一通过 `src/api.ts` 调用后端
- 支持 Tauri 和浏览器双模式
- 浏览器模式使用内存 mock 数据

## 五、构建验证

### 5.1 前端

```bash
npx tsc --noEmit   # TypeScript 类型检查
npx vite build      # Vite 生产构建
```

### 5.2 后端

```bash
cargo check --manifest-path src-tauri/Cargo.toml  # Rust 编译检查
```

### 5.3 完整构建

```bash
npm run build  # tsc + vite build
```
